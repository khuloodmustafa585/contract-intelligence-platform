# Backend Utils Notes 
## 1. file_helpers.py

### Purpose

This file handles reading and saving contract files (PDF, DOCX).

It is the first step in the pipeline:
Upload → Extract Text → Clean → Chunk → AI Analysis

---

### Functions

#### read_pdf(file_path)

* Reads PDF files using PyPDF2
* Loops through all pages
* Extracts text safely (handles None)
* Adds newline between pages for readability

Why important:
PDFs are the main format for contracts, and text extraction must be reliable.

---

#### read_docx(file_path)

* Reads Word documents using python-docx
* Extracts paragraph text
* Joins them into a single string

Why important:
Some contracts are provided as DOCX instead of PDF.

---

#### save_file(file, filename)

* Saves uploaded files into the "uploads" folder
* Ensures directory exists
* Validates file type (.pdf, .docx)
* Uses FastAPI UploadFile (file.file.read())

Why important:
Prevents invalid files and ensures files are stored correctly.

---

### Notes

* Always return empty string "" instead of None to avoid crashes
* file.file.seek(0) resets file pointer before reading
* filename.lower() ensures case-insensitive validation
* uploads/ folder is created automatically if missing

---

## Overall Flow

1. User uploads file
2. save_file() stores it
3. read_pdf() or read_docx() extracts text
4. Text is passed to text_cleaner.py


## 2. text_cleaner.py

### Purpose

This file prepares raw contract text before it is sent to AI.

It is responsible for cleaning and standardizing the text to improve AI accuracy.

Pipeline position:
File Extraction → Text Cleaning → Chunking → AI Analysis

---

### Why It Is Important

Text extracted from PDFs and DOCX files is often messy:

* Extra spaces
* Broken formatting
* Random line breaks
* Inconsistent casing

If this text is sent directly to AI:

* Results will be inaccurate
* Clause detection may fail
* Summaries may be poor

This file ensures the text is clean and consistent before processing.

---

### Functions

#### clean_text(text)

* Removes extra spaces and repeated whitespace
* Removes unwanted special characters
* Keeps important punctuation (.,;:()-)

Why important:
Improves readability and prevents AI confusion caused by noisy text.

---

#### normalize_text(text)

* Converts all text to lowercase

Why important:

* Standardizes input for AI models
* Helps with consistent keyword matching and classification

Note:
Lowercasing is useful for processing, but original text may still be needed for display.

---

#### remove_empty_lines(text)

* Splits text into lines
* Removes empty or whitespace-only lines
* Strips spaces from each line
* Joins lines back together

Why important:
Preserves structure while removing un

## 3. chunking.py

### Purpose

This file is responsible for splitting contract text into smaller parts (chunks) before sending it to the AI.

It ensures that large contracts can be processed efficiently and accurately.

Pipeline position:
Text Cleaning → Chunking → AI Analysis

---

### Why It Is Important

Contracts are usually very long (multiple pages), while AI models have limits on how much text they can process at once.

If the full contract is sent:

* It may exceed token limits
* It becomes slower and more expensive
* AI may lose context or miss important details

Chunking solves this by breaking the contract into manageable pieces.

---

### Functions

#### chunk_text(text, chunk_size)

* Splits text into fixed-size chunks (based on number of words)
* Works by grouping words into blocks

Why important:

* Acts as a fallback method
* Ensures the system works even if the contract has no clear structure

---

#### split_by_clauses(text)

* Uses regex to detect legal sections such as:

  * Article 1
  * Section 2
  * Clause 3
* Splits text based on these headings

Why important:

* Preserves the meaning of the contract
* Each chunk represents a logical legal section

---

#### smart_chunk(text, chunk_size)

* Combines both approaches:

  1. Attempts to split text into clauses
  2. Groups clauses into chunks within a size limit
  3. Falls back to chunk_text if no structure is detected

Why important:

* Produces meaningful chunks (better for AI)
* Ensures chunks do not exceed size limits
* Makes the system more robust and reliable

---

### Key Design Decisions

* Clause-based chunking is preferred over fixed-size chunking:
  Because contracts are structured documents, and meaning is tied to sections.

* Fallback mechanism:
  If clause detection fails, the system still works using basic chunking.

* Chunk size control:
  Prevents sending too much text to AI at once.

---

### How It Connects to the System

1. file_helpers.py extracts raw text from files
2. text_cleaner.py cleans and normalizes the text
3. chunking.py splits the cleaned text into chunks
4. AI processes each chunk separately

---

### Key Takeaways

* Chunking is essential for handling large documents
* Smart chunking improves AI accuracy and efficiency
* Combining structure-based and size-based chunking gives the best results
* This file directly impacts:

  * clause detection
  * risk analysis
  * summaries
  * AI performance

---
### Additional Notes (Important Details)

#### Why do we use `chunk_size = 500`?

* The value `500` represents the maximum size of each chunk (approximate number of words or characters).
* AI models have limits on how much text they can process at once (called token limits).
* If chunks are too large:

  * The AI may fail or truncate input
  * Processing becomes slower and more expensive
* If chunks are too small:

  * The context becomes weak
  * AI may not understand the full meaning

Why 500 specifically:

* It is a balanced default value:

  * Not too big (safe for AI limits)
  * Not too small (keeps enough context)
* It can be adjusted later depending on performance.

---

#### What does this line do?

```python
clauses = re.split(r"(Article \d+|Section \d+|Clause \d+)", text)
```

This line splits the contract text into sections based on common legal headings.

---

#### Explanation of each part:

* `re.split(...)`:

  * A function from the `re` (regular expression) module
  * It splits text whenever a pattern is found

---

* `r"(Article \d+|Section \d+|Clause \d+)"`:

  * This is a **regular expression (regex)** pattern

Breakdown:

* `Article \d+`
  → matches:

  * "Article 1"
  * "Article 2"
  * etc.

* `Section \d+`
  → matches:

  * "Section 1"
  * "Section 10"

* `Clause \d+`
  → matches:

  * "Clause 1"
  * "Clause 5"

* `\d+`
  → means "one or more digits"

* `|` (pipe symbol)
  → means "OR"

So the pattern means:
Split the text whenever you find:
"Article X" OR "Section X" OR "Clause X"

---

#### Important behavior

Because the pattern is inside parentheses `( )`:

* The matched headings (e.g., "Article 1") are INCLUDED in the result
* This is useful because we keep the section titles instead of losing them

---

#### Why this is important

* Contracts are structured documents
* Each clause/section has its own meaning
* Splitting based on these headings allows:

  * Better AI understanding
  * More accurate clause detection
  * Better risk analysis

---

#### Example

Input:
"This agreement... Article 1 Definitions... Article 2 Payment..."

Output:
[
"This agreement...",
"Article 1",
"Definitions...",
"Article 2",
"Payment..."
]

---

#### Key takeaway

This line is what makes chunking "smart" instead of random.

It allows the system to understand contracts in terms of their logical structure rather than just splitting text arbitrarily.
